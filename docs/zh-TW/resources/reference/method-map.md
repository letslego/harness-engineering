# 方法對照表

這個表把最常見的長時編碼代理失敗模式，對應到通常最先能修正它的工件或操作規則。

| 失敗模式 | 實際表現 | 首要修復 | 輔助工件 |
| --- | --- | --- | --- |
| 冷啟動混亂 | 新工作階段把大部分時間花在重新摸索設定與狀態 | 讓儲存庫成為事實來源 | `claude-progress.md` |
| 範圍擴散 | 代理同時啟動多個功能，最後沒有一個完整完成 | 限制目前的活躍範圍 | `feature_list.json` |
| 過早宣稱完成 | 代理改完程式碼就宣稱完成，但沒有可執行證據 | 把完成條件綁到證據 | `clean-state-checklist.md` |
| 啟動脆弱 | 每個工作階段都要重新學一次如何啟動專案 | 標準化設定與驗證 | `init.sh` |
| 交接薄弱 | 下一個工作階段無法判斷哪些內容已驗證、已損壞或接下來該做什麼 | 以明確交接結束 | `session-handoff.md` |
| 評審主觀 | 評審品質取決於個人口味或記憶 | 用固定類別評分輸出 | `evaluator-rubric.md` |

## 操作原則

先加入能直接處理目前失敗模式的最小工件。不要把每一個可靠性問題都丟進同一份全域指令檔案。
